# 别做那么多想当然的假设

原文：[Stop assuming so much](https://nintil.com/dont-assume)

2022-12-15; 最后更新: 2022-12-15

[Ben Kuhn](https://www.benkuhn.net/) 认为，向他人提供人生建议这件事的价值被低估了，我们应该多做这事。本着这种精神，这篇文章就来提点建议。

在做决定时，我们常常把很多事情视为理所当然。大多数时候这没什么问题，因为要求我们在做任何事之前完全不带任何假设是不现实的。如果你出门买面包，你会默认你要去的面包店在营业，默认他们有你想要的面包，并且默认你在去买面包的路上不会遭遇谋杀。原则上，在采取任何行动之前，你确实可以先在谷歌地图上查一下面包店开没开门，或者给他们打个电话确认一下今天有没有黑麦面包。但如果你一天中做的每一件事都要这么确认一遍，那你什么事也干不成。

大多数人生建议都非常宽泛。如果我只是告诉你「别做那么多想当然的假设」，你并不知道到底该在生活中做出什么改变。我并不是说你必须在任何情况下都追求百分之百的确定性；毕竟收集信息是有成本的。那么我想表达的到底是什么呢？我可能是在说，在一些细枝末节上，你通常应该更频繁地去检验你的假设，但这也不完全正确，因为在许多场景下，我们做出的默认假设是完全站得住脚的，不需要时时刻刻去质疑它们。

相反，我想说的是：对于那些你认为很重要的情况（从人际关系到职业发展等一切大事），请专门留出一点时间，去审视一下你在这些情况下做出了哪些假设。然后想一想，把一个「假设」变成「确凿事实」需要付出多大成本。如果看起来划算，那就去验证它。

但光这么说还是不够的，所以我将带你看一系列真实的例子，好让你对我确切的意思有个直观的了解。我也希望这些例子足够令人难忘，以便你在未来做假设时能回想起它们。正如 Ben 所[指出](https://www.benkuhn.net/weeklyessays/)的那样，建议总是听起来容易，做起来难。我们很容易理解为什么做某件事是有益的，但在实际生活中，尽管我们在理性上完全认同这些建议，却往往不会付诸行动。

我猜这是因为建议往往是用非常宽泛的术语给出的：给出建议的人自己已经在践行这条建议了，所以他们只会告诉你他们是怎么做的，却没有给你一个强有力的理由来说明为什么**你**也应该这么做，或者在你刚开始尝试时，该如何说服自己去行动。

我要举的第一个例子就是，有一次我**想当然地认为**一个人一次只能持有一个 [O1 签证](https://nintil.com/us-immigration)（美国杰出人才签证，顺便说一句，我现在持有两个）。我当时知道签证是可以**转移**给新雇主的，我也知道你可以自己成立一家公司雇佣自己，然后让这家公司去跟各方签合同，从而实现同时打几份工。但我认识的人里，没有谁是同时拥有两个 O1 签证的。我甚至都没有试着去网上搜一下这到底可不可行，尽管网上其实有很多信息来源会明确告诉你：同时拥有多个 O1 签证是可行的。直到有一次我和别人聊起这事，对方问我是否**百分之百确定**一个人不能有两个签证时，我当然回答说不知道；然后对方仅仅是让我去核实一下。我心想：「行吧，查一下顶多花 5 秒钟，就算答案是不能同时持有多个 O1 签证也无妨。」于是我给我的律师发了封邮件，没过多久我就收到了确认回复：实际上完全可以同时持有多个 O1 签证。随后我提交了申请，并顺利拿到了我的第二个 O1 签证。

这听起来可能挺蠢的。回想起来，我也觉得当时的自己挺蠢的。去谷歌上搜一下（或者问一下律师）一个人能不能同时持有多个 O1 签证，不过是花 5 秒钟的事；验证成本极低，而潜在的回报（拿到签证）却极高。但在当时的我看来，尽管这现在听起来可能有些令人惊讶，但这就跟「我知道那个路口有一家面包店，没必要看地图确认」是一样的感觉，我对自己的猜测盲目自信了。

关于签证，还有一个类似的例子，那就是我申请我**第一个** O1 签证时的经历：当时看完[申请条件清单](https://www.uscis.gov/policy-manual/volume-2-part-m-chapter-4#3)后，我觉得自己根本达不到要求。但我朋友告诉我，我其实大概率是符合条件的，因为移民局在审查 O1 标准时，并没有字面上写得那么严苛：你必须很优秀，但不一定非得是拿诺贝尔奖那种级别的优秀。在这种情况下，要想「确切知道」自己到底能不能拿到签证，唯一的方法就是去申请（而这是要花钱的）。不过，再次强调，有一个很好的替代方案，那就是去问律师。这时候你可能又会假设去咨询律师肯定要花钱，但现在验证这个假设的成本是零：你只需给律师发封邮件问个问题，看看他们回不回复、收不收费就知道了（答案是不会收费，一般来说，移民律师对这种初次咨询都是免费的）。

当我和 Ben Kuhn 聊起这些例子时，他向我提到了他自己[经常给的一个建议](https://twitter.com/benskuhn/status/1321932187911028736)，那是他在和下属进行 1 对 1 谈话时常说的：如果某人**觉得**另一个人因为某种原因在生他们的气，那么他们应该直接去找那个人谈谈，而不是把这事憋在心里任其发展。因为结果无非是两种：a）这一切都只是他们在胡思乱想（也就是说他们的假设错了，这样一来，任何潜在的冲突就在无形中化解了）；或者 b）那个人确实在生他们的气（他们的假设是对的！那么现在他们就可以坐下来把话说开，讨论到底是因为什么，以及该如何解决问题了）。

在开发 [Rejuvenome](https://astera.org/rejuvenome/) 项目时，我们需要做出的一个关键设计决策是，每个实验队列应该包含多少只小鼠。这不仅取决于所需的统计功效，还取决于我们同时运行多少个队列，以及小鼠的死亡率曲线是什么样的。饲养小鼠的动物房容量大约是 5000 只，所以从纸面上看，研究进度应该是不断增加队列，直到达到 5000 只的上限，然后新招募小鼠的速度只要和小鼠死亡的速度持平就行了。我们制定了一个计划，并且我们大致**想当然地认为**会有足够的空间来实施这个计划。然而，随着订购物流小鼠的日期越来越近，我开始越来越频繁地思考：我们真的有足够的空间来容纳所有这些小鼠吗？最终，我决定用 Python 写一个简单的模拟模型，把小鼠随年龄增长的死亡率因素加进去，然后运行一下看看，在这个计划下，任何给定时间点笼子里到底会有多少只小鼠。结果表明，那个相当雄心勃勃的实验设计根本不可行。如果我们还想测试足够多的队列以验证多种干预措施，我们要么必须缩小实验规模，要么就得找一个更大的动物房。这个结果让人大吃一惊，但紧接着我们就快速重新设计了项目，通过放弃一项要求（即检测微小寿命变化所需的统计功效），我们得以缩小队列的规模，从而把所有小鼠都塞进那个特定的动物房里，最终依然实现了我们想要的大部分目标。幸运的是我们发现得还算早，但这本该更早被发现的：写这个模型并模拟各种场景大概只需要花一天的时间，但它对从招聘到项目成本预算等方方面面产生的影响却是极其巨大的。

为什么会发生这种事？事后看来原因似乎很明显：这个项目有千头万绪的事情要处理，从摸索处理小鼠血液的最佳方案，到搭建数据管道把仪器数据汇总到统一的数据库。我没有在优先级排序上花足够的心思去思考：哪些是今天必须干的，哪些是可以再等一等的。相反，我直接一头扎进了写代码的工作中，因为写代码能带来成就感，而思考优先级不仅耗时更长，还无法提供写代码带来的那种多巴胺刺激。如果我当时问自己一句：「我真的有百分之百的把握，我们能实现我们吹出去的牛吗？」如果诚实作答，我一定会说「没有」。那么，如果接着问「我能做些什么来确保这项研究切实可行？」我就会回答：「去跑个模拟看看实际会发生什么，而不是全凭瞎猜。」最后，如果问自己：「这是目前的头等大事吗？还有更重要的事情要做吗？」我就会意识到这件事必须尽早完成。

发冷邮件是另一个因为「想当然」而付出代价的例子。关于为什么要发冷邮件、如何发，以及如何克服发冷邮件时的心理障碍，[许多](https://colah.github.io/posts/2020-05-University/)[文章](https://guzey.com/personal/what-should-you-do-with-your-life/#cold-emails-and-twitter)都已经探讨过了。经验表明，某个领域的资深人士往往很乐意谈论他们的工作并帮助新人。他们只是不喜欢被人「白嫖」或利用的感觉。这种不愿发邮件打扰别人的心理，其核心就在于此。但是，与其想当然地认为一封经过深思熟虑、措辞得体的邮件会打扰到别人，不如自己先验证一下：如果你自己收到了你正准备发送的那封邮件，你会觉得被打扰了吗？去问问其他经常收到冷邮件的人，他们喜欢帮助别人吗？所有发冷邮件老手总结出的共识就是：冷邮件确实管用，而且一封用心的冷邮件是很受欢迎的，所以别再想当然地以为情况恰恰相反了！

最后一个例子来自特斯拉（Tesla）。在他们的一款车型里有一个零件，某个团队认为装这个零件是出于安全考虑，因此这属于另一个负责安全的团队的管辖范围。但他们不知道的是，第二个安全团队却以为这个零件是为了提供其他某种功能，理应由第一个团队负责。结果就是，这个零件莫名其妙地呆在车上，根本没有任何用处。最终还是马斯克（Elon）直接向这两个团队询问了这个零件的情况，然后把它给拆了。在这个特定的案例中，要逐一核对每一个细小零件的归属权，极其消耗脑力，因此解决办法并不是要求工程师们无时无刻不在做这种核对，而是可以建立一个维护良好的数据库和一套完善的流程，以确保每一个零件都有其存在的正当理由。概括来说，只要让了解相关事实变得更容易、成本更低，人们就更有可能去检验那些想当然的假设，从而犯更少的错。

最后，我们可以把这条建议的逻辑再往前推进一步：有时候我们之所以犯错，就是因为我们想当然的假设太多了。我们可以通过对自己的假设进行更深入的思考来解决这个问题。但具体该怎么做呢？方法就是去思考：**为什么**我们在某些情况下会做出这么多想当然的假设。这里提供几个思路供你检查：

- 更加清楚地意识到为什么你不去验证某个特定的假设。验证一个假设可能需要你去跟某人交谈或发邮件，而你出于某种原因很不情愿去这么做。结果就是，我们自欺欺人地认为这个假设非常可靠，因为这样一来，我们就不必硬着头皮去验证它了。如果情况真是如此，那就把它挑明，写下来：「我的项目现在岌岌可危，仅仅是因为我不愿意去发一封简短的邮件。」如果这句话触动了你，你也许就有勇气把那封邮件发出去了。

- 如果验证某件事需要花费一些精力和时间，那就明确地将这个成本与做错决定的潜在风险进行权衡。想想有没有更简单的验证方法。

- 主动寻求他人的反馈。我们很容易对自己正在做出「想当然的假设」这一事实视而不见，自己很难察觉，但旁人却能一眼看穿。

- 养成定期复盘的习惯。我们平时甚至根本不会专门抽出时间去梳理我们在各个项目中都做出了哪些假设。从理论上讲，这就像在某一天抽出一个小时来进行复盘一样简单，但在实际操作中，这需要你下定决心去养成定期复盘的习惯。

最后一点也是最重要的一点。你可以一边读这篇文章一边频频点头，然后转头就把它抛到九霄云外。或者，你可以从你生活中的某个方面入手，在日历上腾出一个小时，好好想想你在那个方面都做出了哪些想当然的假设。就在我敲下这行字的时候，我自己刚刚就这么做了一遍。这只要花一个小时！今天就可以开始！
